Problem: Hardware orders are being requested frivolously, and because we don’t track the    reason for each request,  we cannot analyze the issue further.

Solution: Alter our Microsoft-SQL database to include reasons and sub reasons for each      hardware request.

But why store only the sub reason ID associated with each hardware order if we wish to also store our primary reason? The relationship between     parent and child reasons is one-to-many, thus any sub reason can only have one parent reason. This way we need only store one additional field.
